# 双检代码评审系统说明书

这套系统服务于一个核心目标：让独立开发者在不依赖人工 Reviewer 的情况下，获得稳定可靠的代码评审。它的策略不是用 AI 替代静态分析，而是让静态规则抓表面问题，让 AI 解释告警并发现静态层漏掉的逻辑 Bug——两条路径互补，合并后给出三路对比。

## 系统目标

1. 快速静态检查：AST + 正则，毫秒级返回表面问题。
2. AI 语义深检：确认告警、排除误报、发现逻辑 Bug。
3. 熔断保护：AI 子系统异常时退化为纯静态报告，不阻断工作流。
4. 模型降级：DeepSeek → Qwen → 本地兜底，提高可用性。
5. CQRS 缓存：同段代码重复评审，第二次直接命中缓存（< 50ms）。
6. 声明式规则：评审阈值和规则放在 YAML 文件中，代码只负责执行。
7. 健康度自检：缓存命中率、熔断器状态、发现质量可度量。
8. 按语言隔离的熔断器池：每个语言独立熔断计数和状态切换。
9. 舱壁隔离：大文件（>500行）走独立线程池（2并发），不阻塞小文件（10并发）。
10. 批量评审：`--batch` 模式支持目录级并发评审，自动检测语言。
11. 向量记忆：Bug 模式自动存入 sqlite，重复出现时匹配历史并给出修复建议。
12. 质量趋势报告：从 CQRS 缓存 + 向量存储生成周报（评分、密度、高频类型、建议）。

## 语言支持

| 语言 | 检查方式 | 规则数 |
|------|---------|--------|
| Python | AST 解析 + 正则 | 7 AST + 2 通用 |
| Java | 正则模式匹配 | 6 专有 + 2 通用 |
| Go | 正则模式匹配 | 3 专有 + 2 通用 |
| JavaScript | 正则模式匹配 | 4 专有 + 2 通用 |

## 评审流程

1. 读取代码内容，自动或手动指定语言。
2. 计算缓存键：`sha256(code + lang + model_ver)`。
3. 查 CQRS 缓存（读路径）→ 命中直接返回。
4. 未命中 → 执行 `static_check(code, lang)`。
5. 检查熔断器 `allow()`：允许→AI降级链，阻断→合并。
6. AI 降级链：DeepSeek → Qwen → 本地兜底。
7. 解析 AI 响应，合并三路对比，计算风险等级。
8. 写缓存，返回 FinalReport。

## 数据契约

### Finding
- `line: int` — 行号
- `kind: str` — 问题类型
- `severity: str` — high / medium / low
- `message: str` — 人类可读描述

### FinalReport
- `risk: str` — 阻止合并 / 修复后合并 / 可合并
- `findings: list[dict]` — 合并后的 finding 列表（含 layer 字段）

## 缓存规则

- 缓存键 = `sha256(code + lang + model_ver)`
- TTL 默认 7 天，`--no-cache` 跳过
- 信封格式 `{"_meta": {...}, "report": {...}}`

## 熔断器规则

- 状态机：CLOSED → OPEN → HALF_OPEN → CLOSED
- 按语言隔离，独立计数和状态切换
- 文件持久化到 `data/.cache/.breaker_state.json`

## 健康度自检

| 维度 | 健康 | 不健康 | 措施 |
| --- | --- | --- | --- |
| 缓存命中率 | >50% | <20% 或膨胀 | 调整 TTL |
| 熔断器状态 | 全部 CLOSED | 频繁 OPEN/振荡 | 检查 API/增大 cooldown |
| 发现质量 | AI 新发现 10-50% | 全确认/全否定 | 调整 prompt/规则 |

## 蒸馏流程

1. 从历史评审记录提取反复出现的 Bug 模式。
2. 写入 `distill/patterns/`，一张卡片一个模式。
3. 卡片含：核心观点、证据、检查方法、触发条件。
4. 跨语言通用写入 `distill/patterns/universal/`。
