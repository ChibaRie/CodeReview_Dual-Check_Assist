# 双检代码评审最佳实践

## 适用场景

双检收益最高的代码：

- 单函数超过 30 行或存在多个逻辑分支
- 非模板、非自动生成的业务代码
- 涉及边界条件、异常处理或并发语义
- 人类容易忽视的偏门写法

对极短的 glue 代码、配置文件或纯数据文件，AI 深检价值有限，
可只跑静态快检。

## 各层能力边界

### 静态快检能做什么

- 函数长度、嵌套深度、圈复杂度
- 可变默认参数、裸 except、`is` 比较字面量
- 超长行、TODO/FIXME 标记

### 静态快检做不到

- 并发/异步语义问题
- 函数参数的业务含义
- 数据流与状态机正确性
- 跨文件的复杂交互

### AI 深检能做什么

- 确认静态层误报或真实问题
- 发现静态层遗漏的语义缺陷
- 给出修复建议和风险评估

### AI 深检的限制

- 需要足够上下文，代码片段过短可能产生幻觉
- 对陌生框架可能误判，建议通过 `--framework` 提供上下文
- 模型输出需解析，偶尔会被代码块标记包裹

## 何时信任哪一层

| 情况                  | 建议                                     |
|-----------------------|------------------------------------------|
| static + ai_confirmed | 高度可信，优先修复                       |
| static + ai_rejection | 可忽略静态误报                           |
| ai_new                | 需要人工复核，防止幻觉                   |
| 仅有 static           | 按风险等级处理；必要时重试 AI            |
| 降级模式              | 至少修复 high 问题，再择机补跑 AI        |

## 提供上下文的技巧

- 用 `--framework` 说明主要框架，帮助 AI 理解惯用写法。
- 对异步/并发代码，在文件名旁加注释说明执行模型。
- 如果代码依赖外部配置或数据库Schema，简单列出关键字段名。

例如：

```bash
python skill/scripts/app.py data/tasks.py --lang python --framework "asyncio + aiohttp"
```

## 风险等级与合并决策

- `阻止合并`：存在 high 严重度问题，必须修复后才能合并。
- `修复后合并`：问题风险可控，修复 medium/low 后可合并。
- `可合并`：当前代码未触发明显风险，可正常合并。

合并前建议至少处理所有 `high` 问题，并对 `ai_new` 做抽样人工确认。
